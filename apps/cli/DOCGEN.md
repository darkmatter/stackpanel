# Documentation Generator

The `stackpanel gendocs` command generates MDX documentation from Nix options JSON and module README files for use with [Fumadocs](https://fumadocs.vercel.app/).

## Usage

```bash
stackpanel gendocs <options.json> <output-dir> [modules-dir]
```

**Arguments:**
- `options.json` - Path to the Nix options JSON file (generated by `nixosOptionsDoc`)
- `output-dir` - Path to the documentation content directory (e.g., `apps/docs/content/docs`)
- `modules-dir` (optional) - Path to scan for module documentation sources

## Output Structure

The command generates:

```
output-dir/
├── reference/       # Options reference documentation
│   ├── index.mdx    # Options index with category links
│   ├── apps.mdx     # Options for stackpanel.apps.*
│   ├── secrets.mdx  # Options for stackpanel.secrets.*
│   └── ...          # One file per category
│
├── modules/         # Module documentation (from README.md/.nix)
│   ├── index.mdx    # Modules index
│   ├── core.mdx     # From nix/stackpanel/core/README.md
│   └── ...          # One file per module
│
└── devenv/          # DevEnv-specific docs (reserved)
```

## Writing Module Documentation

Module documentation can be sourced from two places:

1. **README.md files** - Placed in module directories
2. **.nix file headers** - Multi-line comments at the top of .nix files (5+ lines)

### README.md with Frontmatter

For full control over generated documentation, use YAML frontmatter:

```markdown
---
title: My Module
description: A brief description of what this module does.
icon: Settings
---

# My Module

Your documentation content here...
```

**Supported frontmatter fields:**

| Field | Description | Default |
|-------|-------------|---------|
| `title` | Page title in sidebar and heading | Derived from module name |
| `description` | Meta description for SEO | "Documentation for the X module" |
| `icon` | Lucide icon name for sidebar | None |
| `output` | Custom output path (relative to docs dir) | `modules/{relative-path}.mdx` |
| `skip` | Skip generating docs for this file | `false` |

### Skipping Files

To exclude a README from documentation generation:

```markdown
---
skip: true
---

# Internal Notes

This content won't be generated as docs.
```

### Custom Output Paths

Use the `output` field to place generated docs at a specific location:

```markdown
---
title: Getting Started
description: Quick start guide for Stackpanel.
output: getting-started.mdx
---

# Getting Started

This will be placed at `docs/getting-started.mdx` instead of `docs/modules/...`
```

### Merging Multiple READMEs

Multiple README files can target the same output path. Their content will be concatenated with horizontal rules (`---`) between sections:

```markdown
<!-- nix/stackpanel/core/README.md -->
---
title: Core Concepts
output: concepts/core.mdx
---

# Configuration

Core configuration details...
```

```markdown
<!-- nix/stackpanel/devshell/README.md -->
---
output: concepts/core.mdx
---

# Development Shell

Devshell details get appended...
```

The first README's frontmatter (title, description, icon) takes precedence.

| `AppWindow` | Application definitions |

### README.md without Frontmatter

If no frontmatter is provided, the generator extracts metadata from content:

```markdown
# Module Title

This first paragraph becomes the description.

Rest of content...
```

### .nix File Headers

For simple documentation, add a multi-line comment at the top of your .nix file:

```nix
/*
  Module Title

  This is the description that will be extracted.

  Additional documentation content goes here.
  Can span multiple paragraphs.
*/
{ config, lib, pkgs, ... }:
{
  # Module implementation
}
```

Requirements:
- Must be at the very top of the file
- Must be 5+ lines (to distinguish from simple comments)
- First non-empty line becomes the title
- Subsequent content becomes description/body

### @docgen Directives for .nix Files

Use `@docgen.*` directives on their own line within nix comments to control documentation generation:

```nix
# Module Title
#
# @docgen.icon Lock
# @docgen.output reference/secrets.mdx
#
# This is the description and content.
# The directives above are stripped from the output.
{ config, lib, ... }:
```

**Available directives:**

| Directive | Description |
|-----------|-------------|
| `@docgen.skip` | Skip generating docs for this file |
| `@docgen.icon IconName` | Set the Lucide icon for this doc |
| `@docgen.output path/to/file.mdx` | Set custom output path |

Example with skip:

```nix
# Internal Implementation Details
#
# @docgen.skip
#
# This file contains internal implementation notes
# that should not be published to documentation.
{ ... }:
```

## Example Workflow

1. **Write module documentation:**
   ```markdown
   <!-- nix/stackpanel/secrets/README.md -->
   ---
   title: Secrets Management
   description: Configure encrypted secrets with SOPS and vals integration.
   icon: Lock
   ---

   # Secrets Management

   Stackpanel provides secure secrets management using SOPS encryption...
   ```

2. **Generate documentation:**
   ```bash
   # Generate options JSON first (from Nix)
   nix build .#options-json

   # Run docgen
   stackpanel gendocs ./result/options.json ./apps/docs/content/docs ./nix/stackpanel
   ```

3. **Review generated files:**
   ```
   ✓ apps/docs/content/docs/reference/index.mdx
   ✓ apps/docs/content/docs/reference/secrets.mdx (5 options)
   ✓ apps/docs/content/docs/modules/secrets.mdx (README)
   ```

## Integration with Fumadocs

The generated MDX files are compatible with Fumadocs out of the box:

- Frontmatter fields (`title`, `description`, `icon`) are recognized by Fumadocs
- MDX content supports all Fumadocs components
- Links between pages work with relative paths

## Troubleshooting

**No modules found:**
- Ensure README.md files exist in module directories
- Check that .nix files have 5+ line comment headers

**Icons not showing:**
- Verify icon name matches a [Lucide icon](https://lucide.dev/icons) exactly (PascalCase)
- Ensure Fumadocs has Lucide icons configured

**Frontmatter not parsed:**
- Frontmatter must start on line 1 with `---`
- Closing `---` must be on its own line
- Use proper YAML syntax (no tabs, consistent spacing)
