Stackpanel can generate Fly.io deployment configuration from your project definition—`fly.toml` files, Dockerfiles, and OIDC-based authentication for passwordless deploys from CI. Instead of hand-maintaining Fly config that drifts from your actual app setup, the deployment extension derives everything from your Stackpanel config.

## Enable Deployment

Deployment configuration is per-app. You don't need a global "Fly extension" - just configure your apps with `deployment.host = "fly"`.

Stackpanel will automatically add the `flyctl` CLI to your devshell when you have apps configured for Fly.io deployment.

## Defining Fly Apps

Each app that should be deployed to Fly needs deployment configuration:

```nix
stackpanel.apps = {
  api = {
    port = 1;
    root = "./apps/api";
    build = "bun run build";

    deployment = {
      enable = true;
      host = "fly";
      
      fly = {
        appName = "myapp-api";
        region = "iad";
        memory = "512mb";
        cpus = 1;
        forceHttps = true;
      };
    };
  };

  web = {
    port = 0;
    root = "./apps/web";
    build = "bun run build";

    deployment = {
      enable = true;
      host = "fly";
      
      fly = {
        appName = "myapp-web";
        region = "iad";
      };
    };
  };
};
```

Stackpanel generates deployment configuration for each app, wiring up the correct ports, health checks, and resource settings.

## Generated Files

When the Fly extension is enabled, Stackpanel generates:

| File | Description |
|------|-------------|
| `apps/<app>/fly.toml` | Fly.io app configuration |
| `apps/<app>/Dockerfile` | Container image (if not already defined) |

These are standard Fly.io files—`flyctl` reads them directly. Like all Stackpanel-generated files, they're committed to Git so that deploys work without Nix.

```nix
# The extension generates fly.toml from your config
stackpanel.files.entries."apps/api/fly.toml" = {
  # ... generated automatically from deployment.fly settings
};
```

<Callout type="info">
You can inspect the generated `fly.toml` at any time. It's a standard file that `flyctl` understands—no proprietary format or wrapper.
</Callout>

## Container Builds

Fly.io deploys container images. Configure containers alongside your app deployment:

```nix
stackpanel.apps.api = {
  port = 1;
  root = "./apps/api";
  build = "bun run build";

  deployment = {
    enable = true;
    host = "fly";
    
    fly = {
      appName = "myapp-api";
      region = "iad";
    };
  };

  container = {
    enable = true;
    entrypoint = [ "${pkgs.bun}/bin/bun" "run" "dist/index.js" ];
  };
};
```

Or define containers separately using the [containers module](/docs/deployment/containers) with matching names.

## OIDC Authentication

For CI/CD deployments, configure OIDC authentication at the global deployment level:

```nix
stackpanel.deployment = {
  fly = {
    organization = "my-fly-org";
  };
};
```

When configured, the [CI generation](/docs/apps-ci/ci-generation) can produce GitHub Actions steps that authenticate with Fly using GitHub's OIDC token. This means:

- No `FLY_API_TOKEN` secret to manage
- Tokens are short-lived and scoped to each CI run
- Authentication is tied to your GitHub org/repo identity
- No token rotation or expiration to worry about

```yaml
# Generated CI step (simplified)
- name: Deploy to Fly
  uses: superfly/flyctl-actions/setup-flyctl@master
- run: flyctl deploy --remote-only
  env:
    FLY_API_TOKEN: ${{ steps.fly-oidc.outputs.token }}
```

<Callout type="warn">
OIDC authentication requires your Fly.io organization to have OIDC configured with your GitHub repository. See the [Fly.io OIDC docs](https://fly.io/docs/reference/openid-connect/) for setup instructions.
</Callout>

## Multi-Region Deployment

Deploy to multiple Fly.io regions for lower latency:

```nix
stackpanel.apps.api.deployment.fly = {
  appName = "myapp-api";
  region = "iad";
  
  # Note: Multi-region configuration is managed through Fly.io directly
  # Set your primary region here and use flyctl to scale to additional regions
};
```

After deploying to your primary region, use `flyctl scale` to add machines in additional regions.

## Environment and Secrets

Configure environment variables and secrets for your Fly.io deployments:

```nix
stackpanel.apps.api.deployment.fly = {
  appName = "myapp-api";

  # These get set as Fly secrets
  secrets = [
    "DATABASE_URL"
    "API_KEY"
    "STRIPE_SECRET_KEY"
  ];

  # Static environment variables (non-secret)
  env = {
    NODE_ENV = "production";
    LOG_LEVEL = "info";
  };
};
```

Environment variables are included in deployment configuration. Secrets are managed through Stackpanel's [secrets system](/docs/secrets/variables) and synced to Fly.io.

## Deployment Scripts

The Fly extension registers scripts for common deployment tasks:

```bash
# Deploy a specific app
fly:deploy api

# Deploy all Fly-configured apps
fly:deploy-all

# Check deployment status
fly:status

# Open the Fly dashboard
fly:dashboard
```

These are standard Stackpanel [scripts](/docs/dev-environment/packages-scripts) that wrap `flyctl` with the correct app names and configuration.

## Databases on Fly

For managed databases, use Fly's CLI to create Postgres instances, then reference them in your app:

```bash
# Create a Postgres cluster
flyctl postgres create --name myapp-db --region iad

# Attach it to your app
flyctl postgres attach myapp-db --app myapp-api
```

Then reference the database in your deployment config:

```nix
stackpanel.apps.api.deployment.fly = {
  appName = "myapp-api";
  
  secrets = [
    "DATABASE_URL"  # Automatically set by Fly when attached
  ];
};
```

<Callout type="info">
For local development, use Stackpanel's built-in [PostgreSQL service](/docs/services/postgres) instead of Fly Postgres.
</Callout>

## Combining with Other Deployment Targets

You can deploy different apps in your monorepo to different platforms:

```nix
stackpanel.apps = {
  web = {
    port = 0;
    # Deploy the frontend to Cloudflare
    deployment = {
      enable = true;
      host = "cloudflare";
    };
  };

  api = {
    port = 1;
    # Deploy the API to Fly.io
    deployment = {
      enable = true;
      host = "fly";
      fly.appName = "myapp-api";
    };
  };
};
```

Each deployment target generates its own configuration independently. The [CI generation](/docs/apps-ci/ci-generation) produces workflow steps for all enabled deployment targets.

## CLI Commands

When you have apps configured for Fly.io deployment, `flyctl` is automatically added to your devshell:

```bash
# Direct flyctl access
flyctl status --app myapp-api
flyctl logs --app myapp-api
flyctl ssh console --app myapp-api

# Build containers for deployment
nix build .#packages.x86_64-linux.container-api
```

## Troubleshooting

### "App not found" errors

Make sure the Fly app exists. On first deployment, you may need to create it:

```bash
flyctl apps create myapp-api --org my-org
```

Or let `flyctl deploy` create it automatically on the first run.

### Build failures in CI

Check that the generated Dockerfile is correct:

```bash
cat apps/api/Dockerfile
```

If the auto-generated Dockerfile doesn't work for your app, you can provide a custom one and point the extension at it.

### OIDC token errors

Verify that your Fly.io organization has OIDC configured for your GitHub repository. The OIDC subject must match what Fly expects—typically `repo:<org>/<repo>:ref:refs/heads/main` or similar.

```bash
# Test OIDC locally (requires flyctl >= 0.1.130)
flyctl auth token --oidc
```

### Secrets not available in the deployed app

Fly secrets are set separately from `fly.toml` environment variables. Make sure secrets are actually set:

```bash
flyctl secrets list --app myapp-api
```

If a secret is missing, set it manually or re-run the deployment script that syncs Stackpanel variables to Fly secrets.
