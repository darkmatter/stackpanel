This directory contains **centralized** option definitions for the Stackpanel module system -- specifically options that are consumed or produced by multiple modules.

Single-consumer options are colocated with their implementation modules (see below).

## Overview

### Centralized options (in `core/options/`)

These options are consumed by multiple modules and must remain centralized:

| File | Description | Consumers |
|------|-------------|-----------|
| `core.nix` | Root paths, directories, basic settings | Many |
| `apps.nix` | Application port and domain configuration | Many |
| `ports.nix` | Deterministic port computation | Many |
| `devshell.nix` | Shell environment (packages, hooks, commands, files) | Many |
| `global-services.nix` | PostgreSQL, Redis, Minio, Caddy services | Many |
| `caddy.nix` | Caddy reverse proxy configuration | Multiple |
| `aws.nix` | AWS Roles Anywhere certificate auth | 6 modules |
| `step-ca.nix` | Step CA certificate management | 5 modules |
| `ide.nix` | VS Code, Zed, Cursor integration | Multiple |
| `secrets.nix` | SOPS-encrypted secrets management | Multiple |
| `codegen.nix` | Code generator definitions | Multiple |
| `motd.nix` | Message of the Day help display | 13+ producers |
| `variables.nix` | Variable data bus | Multiple producers/consumers |
| `variables-backend.nix` | Secrets backend config | 6 modules |
| `tasks.nix` | Task definitions | Multiple producers/consumers |
| `users.nix` | User identity schema | Codegen + IDE |

### Colocated options (in implementation modules)

These options have a single consumer and are defined alongside their implementation:

| Option | Location | Why colocated |
|--------|----------|---------------|
| `stackpanel.dns` | `network/dns.nix` | Only used by DNS module |
| `stackpanel.binary-cache` | `services/binary-cache.nix` | Only used by binary-cache module |
| `stackpanel.theme` | `tui/theme.nix` | Only used by theme module |
| `stackpanel.cli` | `core/cli.nix` | Only used by CLI module |
| `stackpanel.state` | `core/state.nix` | Only used by state module |
| `stackpanel.ci` | `apps/ci.nix` | Only used by CI module |

## Option Namespaces

All options are under the `stackpanel` namespace:

```nix
stackpanel.enable           # Master switch
stackpanel.root             # Project root path
stackpanel.dirs.*           # Directory configuration
stackpanel.apps.*           # Application definitions
stackpanel.ports.*          # Port configuration
stackpanel.devshell.*       # Shell environment
stackpanel.globalServices.* # Development services
stackpanel.network.*        # Network/TLS settings
stackpanel.secrets.*        # Secrets management
stackpanel.ide.*            # IDE integration
stackpanel.cli.*            # CLI settings
stackpanel.motd.*           # Help message
stackpanel.theme.*          # Prompt theming
```

## Design Principles

### Pure Declarations (centralized options)

Centralized option files in `core/options/` contain only `lib.mkOption` declarations. No implementation logic, no `config = ...` blocks. This allows the schema to be:
- Evaluated without side effects
- Used for documentation generation
- Validated independently of implementation

Colocated options live in the same file as their `config = ...` implementation, using the standard NixOS module pattern of `{ options = ...; config = ...; }`.

### Adapter-Agnostic

Options work with any Nix module system:
- devenv
- NixOS modules
- flake-parts
- `lib.evalModules`

Implementation is provided by adapter modules that translate options to the target system.

### Hierarchical Types

Complex options use `lib.types.submodule` for nested configuration:

```nix
options.stackpanel.apps = lib.mkOption {
  type = lib.types.attrsOf (lib.types.submodule {
    options = {
      offset = lib.mkOption { ... };
      domain = lib.mkOption { ... };
      tls = lib.mkOption { ... };
    };
  });
};
```

### Computed Values

Read-only options expose computed values:

```nix
options.stackpanel.ports.base-port = lib.mkOption {
  type = lib.types.port;
  readOnly = true;
  default = basePort;  # Computed from project-name
};
```

## Adding New Options

**Single-consumer option (preferred):** Define the option directly in the implementation module. No need to touch `core/options/` at all.

```nix
# nix/stackpanel/mymodule/myfeature.nix
{ lib, config, pkgs, ... }:
let cfg = config.stackpanel.myfeature;
in {
  options.stackpanel.myfeature = {
    enable = lib.mkEnableOption "My feature description";
    setting = lib.mkOption {
      type = lib.types.str;
      default = "default-value";
      description = "What this setting controls";
    };
  };

  config = lib.mkIf cfg.enable {
    # Implementation here
  };
}
```

**Multi-consumer option:** Create a separate file in `core/options/` and import it in `default.nix`:

1. Create `core/options/<feature>.nix` with only `options.stackpanel.<feature> = ...`
2. Import the file in `core/options/default.nix`
3. Implement consumers in separate modules that read `config.stackpanel.<feature>`
