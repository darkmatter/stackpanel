Variables are the primary way to define configuration values that get exported as environment variables in your dev shell. They support multiple types, ranging from plain text to encrypted secrets.

## Variable Types

| Type | Description | Resolved At |
|------|-------------|-------------|
| LITERAL | Plain text value (stored in `value` field) | Nix eval time |
| SECRET | Encrypted in SOPS group file (`value` is empty) | Runtime (agent) |
| EXEC | Shell command that produces the value | Runtime (agent) |

The variable type is determined by the **ID prefix** (keygroup), not the value:

| ID Prefix | Type | Storage |
|-----------|------|---------|
| `/var/*` | Config (literal) | `value` field in Nix config |
| `/dev/*`, `/prod/*`, `/staging/*`, `/common/*` | Secret | SOPS file `vars/<group>.sops.yaml` |
| `/computed/*` | Computed | Derived at eval time |

## Defining Variables

Variables are defined under stackpanel.variables with a path-style key:

```nix
stackpanel.variables = {
  "/dev/postgres-url" = {
    key = "POSTGRES_URL";
    type = "LITERAL";
    value = "postgresql://localhost:5432/dev";
  };

  "/prod/api-key" = {
    key = "API_KEY";
    type = "SECRET";
    master-keys = [ "prod" ];
  };

  "/shared/openai-key" = {
    key = "OPENAI_API_KEY";
    type = "SECRET";
    master-keys = [ "dev" "prod" ];
  };

  "/dev/git-commit" = {
    key = "GIT_COMMIT";
    type = "EXEC";
    value = "git rev-parse --short HEAD";
  };
};
```

The path prefix (e.g., /dev/, /prod/) is a convention for organizing variables by environment. It is not enforced but helps keep things tidy.

## Managing Variables via CLI

```bash
# Set a literal value
stackpanel vars set DATABASE_URL --value "postgresql://localhost:5432/dev"

# Set an encrypted secret
stackpanel vars set API_KEY --value "sk-abc123" --secret --keys dev,prod

# Get a value (decrypts if needed)
stackpanel vars get API_KEY

# List all variables
stackpanel vars list

# Delete a variable
stackpanel vars delete API_KEY
```

## LITERAL Variables

The simplest type. The value is set directly in Nix config and baked into the shell at eval time:

```nix
"/dev/app-name" = {
  key = "APP_NAME";
  type = "LITERAL";
  value = "myapp";
};
```

Use LITERAL for values that are not sensitive and do not change between machines, such as app names, feature flags, and default URLs.

## SECRET Variables

Encrypted in SOPS group files. The variable's `value` field is an empty string -- the actual secret value lives only in the SOPS file.

```nix
"/prod/stripe-key" = {
  key = "STRIPE_SECRET_KEY";
  value = "";  # actual value in vars/prod.sops.yaml
};
```

Set the value via CLI:
```bash
secrets:set stripe-key --group prod --value "sk_live_xxx"
```

At runtime, the agent decrypts the value directly from `vars/prod.sops.yaml` using the group's AGE key. See [Master Keys](./master-keys) for how encryption keys are managed.

## EXEC Variables

Run a shell command to produce the value dynamically:

```nix
"/dev/git-commit" = {
  key = "GIT_COMMIT";
  type = "EXEC";
  value = "git rev-parse --short HEAD";
};

"/dev/timestamp" = {
  key = "BUILD_TIMESTAMP";
  type = "EXEC";
  value = "date -u +%Y-%m-%dT%H:%M:%SZ";
};
```

The command runs at shell entry time, and the output becomes the variable value.

<Callout type="warn">
EXEC variables run on every shell entry. Keep the commands fast and side-effect-free. Avoid commands that make network calls or modify state.
</Callout>

## Variable Resolution Order

When you enter the dev shell, variables are resolved in this order:

1. LITERAL values are set immediately during Nix evaluation
2. EXEC commands run and their output is captured
3. SECRET values are decrypted by the agent
4. VALS references are resolved by the agent

If a variable cannot be resolved (e.g., the master key is unavailable or a vals reference fails), the agent logs a warning but continues. Your shell still starts with whatever variables could be resolved.

## Per-User Overrides

Variables defined in config.local.nix (gitignored) override shared variables:

```nix
# .stackpanel/config.local.nix
{
  stackpanel.variables."/dev/postgres-url".value = "postgresql://localhost:5432/my_local_db";
}
```

This lets individual developers customize connection strings and other values without affecting the team.
