Master keys are AGE encryption keys used to encrypt and decrypt secrets. Each "group" (e.g., `dev`, `prod`) has its own keypair. The public key lives in `config.nix`, the private key is SOPS-encrypted and committed to git.

## Setup: Initialize a Group

Before you can store secrets, you need at least one initialized group.

### 1. Run init-group

```bash
secrets:init-group dev
```

This generates an AGE keypair and outputs the public key. You'll see output like:

```
Generating AGE keypair for group: dev
Encrypting private key to recipients/dev.enc.age...
Uploading group key to GitHub Actions secret SECRETS_AGE_KEY_DEV...
Generating rekey workflow at .github/workflows/secrets-rekey.yml...

Group 'dev' initialized successfully.

  Public key: age18jexhwcd4ehm7th0pqt8x6rmg3yuxstwv8aavqul4hvcmjkkhfaqn75su0

Next steps:
  1. Add to .stack/config.nix:
     stackpanel.secrets.groups.dev.age-pub = "age18jex...";
  2. Re-enter the devshell: exit && nix develop --impure
  3. Commit: git add .stack/secrets/ .github/workflows/ && git commit
```

### 2. Add the public key to config.nix

Copy the printed `age-pub` value and add it to your config:

```nix
# .stack/config.nix
stackpanel.secrets.groups.dev.age-pub = "age18jexhwcd4ehm7th0pqt8x6rmg3yuxstwv8aavqul4hvcmjkkhfaqn75su0";
```

### 3. Re-enter the devshell

The config change only takes effect after re-entering:

```bash
exit
nix develop --impure
```

### 4. Set a secret to verify

```bash
secrets:set DATABASE_URL --group dev --value "postgresql://localhost:5432/mydb"
secrets:get DATABASE_URL --group dev
# Output: postgresql://localhost:5432/mydb
```

### 5. Commit

```bash
git add .stack/secrets/ .github/workflows/secrets-rekey.yml
git commit -m "Initialize dev group secrets"
git push
```

<Callout type="info">
You can repeat this process for other groups like `prod` or `staging`. Each group has its own keypair and can hold different secrets.
</Callout>

### `secrets:init-group` Flags

| Flag | Description |
|------|-------------|
| `--ssm-path <path>` | Custom AWS SSM path for the private key |
| `--no-ssm` | Skip AWS SSM storage |
| `--no-gh` | Skip GitHub secret upload and workflow generation |
| `--force-gh` | Overwrite an existing GitHub secret (use with care) |
| `--dry-run` | Preview without making changes |
| `--yes` / `-y` | Skip confirmation prompts |
| `--json` | Output results as JSON |

## How Keys Work

### The encryption chain

```
config.nix  (group public key — single source of truth)
    ↓ used by wrapped SOPS for encryption
vars/dev.sops.yaml  (SOPS-encrypted secrets)
    ↓ decrypted using group private key
recipients/dev.enc.age  (group private key, SOPS-encrypted to all recipients)
    ↓ decrypted using your local key
.stack/keys/local.txt  (your local AGE private key, gitignored)
```

When you run `secrets:get`, the chain is traversed bottom-up: your local key decrypts the group private key, which decrypts the group YAML.

### Local key (auto-generated)

Every developer gets a local AGE key auto-generated on first shell entry:

- **Private key**: `.stack/keys/local.txt` (gitignored)
- **Public key**: `.stack/keys/local.pub` (gitignored)

To register yourself as a recipient (so you can decrypt group keys), run:

```bash
secrets:join              # Join the default "team" group
secrets:join --group admins  # Join the admins group
```

This copies your public key to `recipients/team/<username>.age.pub` (committed to git).

### Wrapped SOPS

A wrapped `sops` binary is provided in the devshell that automatically:
- Reads group public keys from `config.nix` (baked in at Nix eval time)
- Injects the correct `--age` recipient when encrypting group files
- Sets `SOPS_AGE_KEY_CMD` for automatic private key resolution

This means `sops edit vars/dev.sops.yaml` just works -- no need to remember key flags.

### Decrypting group keys manually

If `SOPS_AGE_KEY_CMD` can't auto-resolve a group key (e.g., after a fresh clone before the rekey workflow runs), you can decrypt group keys manually:

```bash
.stack/secrets/bin/decrypt.sh dev        # decrypt a specific group
.stack/secrets/bin/decrypt.sh --all      # decrypt all groups
```

This writes a plaintext `.age` file next to the `.enc.age` file (gitignored). Once decrypted, `SOPS_AGE_KEY_CMD` picks it up automatically.

### Generated `.sops.yaml`

The `vars/.sops.yaml` file is generated from `config.nix` on every shell entry and is gitignored. It serves as a fallback for raw `sops` calls. The canonical key source is always `config.nix`.

## Team Onboarding

New team members can onboard themselves without help from existing members, as long as they have write access to the repository.

### The flow

1. **Clone and enter devshell**: `nix develop --impure`
   - A local AGE key is auto-generated

2. **Join the team**: `secrets:join`
   - Copies your public key to `recipients/team/<username>.age.pub`

3. **Commit and push the `.pub` file**:
   ```bash
   git add .stack/secrets/recipients/team/
   git commit -m "Add my recipient key"
   git push
   ```

4. **GitHub Actions rekey workflow triggers**:
    - Collects all public keys from `recipients/team/` and `recipients/admins/`
    - Re-encrypts all `.enc.age` files for ALL recipients
    - Commits and pushes the result

5. **Pull the re-encrypted files**:
    ```bash
    git pull
    ```
    Now they can decrypt all secrets.

### Recipients directory

Every developer's public key is tracked in a committed directory, organized by recipient group:

```
.stack/secrets/recipients/
  team/
    alice.age.pub       # Alice's AGE public key
    bob.age.pub         # Bob's AGE public key
    carol.ed25519.pub   # Carol's SSH ED25519 key (converted to AGE)
  admins/
    alice.age.pub       # Admin access
```

The recipient name defaults to your git username, then system username. SSH ED25519 keys are supported alongside native AGE keys.

To add a recipient without the devshell (e.g., from CI or a non-Nix environment):

```bash
.stack/secrets/bin/add-recipient.sh --key "age1abc..." --name alice --group team
.stack/secrets/bin/add-recipient.sh --key "ssh-ed25519 AAAA..." --name bob --group admins
```

### The rekey workflow

The generated `.github/workflows/secrets-rekey.yml`:
- Triggers when `recipients/**/*.age.pub` files are pushed
- Uses the group private key (from GitHub Actions secrets) to decrypt `.enc.age` files
- Re-encrypts them for all current recipients
- Commits and pushes

<Callout type="warn">
By default, `secrets:init-group` will **not** overwrite an existing GitHub Actions secret. This prevents accidentally breaking the rekey workflow. Use `--force-gh` only when you intentionally want to replace the key.
</Callout>

## External Secret Stores

Group private keys can optionally be stored in external backends for CI/CD environments. Two approaches:

### Via `.vals` files

Place a `.vals` file in `recipients/` with a reference:

```
# recipients/prod.vals
ref+awsssm:///stackpanel/keys/prod
```

### Via `key-cmd` option

```nix
stackpanel.secrets.groups.prod.key-cmd = "op read 'op://vault/stackpanel/age-key'";
```

Note: `vals` is only used for group key resolution from external stores. Secret values are decrypted directly from SOPS files -- no `ref+sops://` indirection.

## Key Rotation

Use the standalone `rotate.sh` script to rotate a group's master key. It handles the full lifecycle: archiving the old key, generating a new keypair, re-encrypting the group's SOPS file, and updating the `.enc.age` file.

```bash
.stack/secrets/bin/rotate.sh dev
```

This will:
1. Archive the old private key to `recipients/.archive/`
2. Generate a new AGE keypair
3. Write new `.age` and `.age.pub` files
4. Encrypt the new private key as `.enc.age` (to all recipients)
5. Re-encrypt `vars/dev.sops.yaml` to the new key

After rotation:

1. Update `config.nix` with the new public key (printed by the script)
2. Re-enter the devshell
3. Commit and push
4. Optionally run `secrets:init-group dev --force-gh` to update the GitHub Actions secret with the new key
