Stackpanel provides encrypted secrets management built on [SOPS](https://github.com/getsops/sops) and [AGE](https://github.com/FiloSottile/age). Secrets are stored directly in SOPS-encrypted YAML files organized by group (e.g., `dev`, `prod`, `common`). Variable IDs encode the group and key (`/dev/MY_KEY`), and the actual secret values live only in the SOPS files -- not in Nix config. Team access is managed through a self-service git-based flow.

## Getting Started

Follow these steps to set up secrets for your project. If you just want to store secrets locally during development, you only need Step 1. For team sharing, complete all steps.

### Step 1: Enter the devshell

```bash
nix develop --impure
```

This automatically generates your local AGE key. To register yourself as a team recipient, run `secrets:join` (see [Team Collaboration](#team-collaboration)).

### Step 2: Initialize a group

Groups are how secrets are organized (typically by environment). Initialize the `dev` group:

```bash
secrets:init-group dev
```

This will:
1. Generate an AGE keypair for the group
2. SOPS-encrypt the private key (committed to git as `keys/dev.enc.age`)
3. Upload the group private key to GitHub Actions (for team rekey workflow)
4. Generate `.github/workflows/secrets-rekey.yml`
5. Print the group's public key

<Callout type="warn">
Copy the printed `age-pub` value. You need it for the next step.
</Callout>

### Step 3: Add the group key to config

Open `.stackpanel/config.nix` and add the public key:

```nix
stackpanel.secrets.groups.dev.age-pub = "age1...paste-the-key-here...";
```

Then re-enter the devshell so the config takes effect:

```bash
exit
nix develop --impure
```

### Step 4: Set your first secret

```bash
secrets:set MY_API_KEY --group dev --value "sk-abc123"
```

### Step 5: Verify it works

```bash
# Retrieve the secret
secrets:get MY_API_KEY --group dev

# List all secrets
secrets:list
```

### Step 6: Commit everything

```bash
git add .stackpanel/secrets/ .github/workflows/secrets-rekey.yml
git commit -m "Initialize dev group secrets"
git push
```

Your secrets are now encrypted at rest in git. Team members can onboard themselves (see [Team Collaboration](#team-collaboration)).

## How It Works

Secrets are stored in **SOPS-encrypted group YAML files** (e.g., `.stackpanel/secrets/vars/dev.sops.yaml`). Each group has its own AGE keypair. Variable IDs encode the group: `/dev/MY_KEY` stores its value under key `MY_KEY` in `vars/dev.sops.yaml`.

**Setting a secret:**
1. `sops set` writes the key/value into the group's YAML file (non-destructive)
2. The wrapped SOPS looks up the group's AGE public key from `config.nix`
3. The encrypted file is committed to git
4. The variable in `config.nix` has `value = ""` -- the SOPS file is the source of truth

**Getting a secret:**
1. Your local key decrypts the group's private key (`recipients/dev.enc.age`)
2. The group private key decrypts the group YAML file via SOPS
3. The specific value is extracted and returned

### Key integrity

Group public keys live exclusively in `config.nix`. A **wrapped SOPS** binary automatically injects the correct `--age` recipient per group, and `vars/.sops.yaml` is generated at shell entry from the same config (never checked in). This prevents key drift.

## CLI Commands

| Command | Description |
|---------|-------------|
| `secrets:set KEY --group GROUP --value VALUE` | Set a secret in a group |
| `secrets:get KEY --group GROUP` | Decrypt and retrieve a secret |
| `secrets:list [GROUP]` | List all secrets (optionally by group) |
| `secrets:load GROUP [--format dotenv\|json\|yaml]` | Decrypt a group's SOPS file |
| `secrets:join [--group GROUP]` | Add your local key to a recipient group |
| `secrets:init-group <name>` | Initialize a group keypair and set up team access |
| `secrets:show-keys` | Show master keys and groups with status |
| `secrets:rekey` | Re-encrypt a secret to different master keys |

## Standalone Scripts (`bin/`)

The `.stackpanel/secrets/bin/` directory contains shell scripts that work **without the Nix devshell** -- they only need `sops`, `age`, and `jq` in PATH. These are useful for CI/CD pipelines, bootstrapping, and environments where Nix isn't available.

| Script | Description | Dependencies |
|--------|-------------|--------------|
| `rekey.sh [group]` | Re-encrypt `.enc.age` files to all current recipients. Run after adding a new team member's public key. | `sops` |
| `rotate.sh <group>` | Rotate a group's AGE keypair: archives the old key, generates a new one, re-encrypts the group's SOPS file. | `age`, `sops` |
| `decrypt.sh <group>\|--all` | Decrypt a group's `.enc.age` to a plaintext `.age` file for local secret resolution. The `.age` file is gitignored. | `sops`, `age` |
| `add-recipient.sh --key <key> --name <name> [--group <group>]` | Add an AGE or SSH ED25519 public key to a recipient group directory. | none |
| `codegen.sh [--app <name>]` | Generate typed TypeScript env modules from `apps.json` and `vars/*.sops.yaml`. | `sops`, `jq` |

Example usage:

```bash
# After a new team member pushes their .age.pub file
.stackpanel/secrets/bin/rekey.sh

# Rotate the dev group key (generates new keypair, re-encrypts everything)
.stackpanel/secrets/bin/rotate.sh dev

# Decrypt all group keys locally
.stackpanel/secrets/bin/decrypt.sh --all

# Add a recipient without the devshell
.stackpanel/secrets/bin/add-recipient.sh --key "age1abc..." --name alice --group team
```

<Callout type="info">
These scripts are the same operations that the devshell `secrets:*` commands perform, but packaged as portable shell scripts. Use the devshell commands during normal development, and the `bin/` scripts when the devshell isn't available.
</Callout>

## Team Collaboration

New team members can onboard themselves with minimal help:

1. Clone the repo and enter the devshell (`nix develop --impure`) -- a local AGE key is auto-generated
2. Run `secrets:join` to register their public key in `recipients/team/`
3. Commit and push the `.age.pub` file
4. A GitHub Actions workflow triggers and re-encrypts all group keys for the new recipient
5. Pull the result -- they can now decrypt all secrets

See [Master Keys - Team Onboarding](/docs/secrets/master-keys#team-onboarding) for the full guide.

## External Secret Stores

Group private keys can optionally be stored in external backends. This is useful for CI/CD where the `.enc.age` file isn't available. Store a `.vals` file in `recipients/` with a reference that resolves to the AGE private key:

| Store | Reference Format |
|-------|-----------------|
| AWS SSM Parameter Store | `ref+awsssm://path/to/key` |
| HashiCorp Vault | `ref+vault://secret/data/path#key` |
| GCP Secret Manager | `ref+gcpsecrets://project/secret-name` |

Alternatively, use the `key-cmd` option per group for custom resolution commands (e.g., 1Password CLI).

Note: `vals` is only needed for group key resolution from external stores. Secret values are decrypted directly from SOPS files -- no `ref+sops://` indirection.
