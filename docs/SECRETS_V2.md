# Secrets v2: Direct SOPS Architecture

## Motivation

The current secrets system routes every secret value through `vals` references
(`ref+sops://.stackpanel/secrets/vars/dev.sops.yaml#/KEY`). This adds an
indirection layer that:

- Makes SOPS files non-portable (consumers need `vals`, not just `sops`)
- Creates N vals resolutions per app instead of 1 `sops decrypt` per group
- Puts implementation details (full file paths) into user-facing config
- Makes the variable system tightly coupled to the resolution mechanism

The v2 architecture makes SOPS files the primary interface. `sops decrypt`
gives you your secrets directly. `vals` becomes optional for edge cases
(external secret stores, mixed backends).

## Design Principles

1. **SOPS files are portable artifacts** -- `sops decrypt vars/dev.sops.yaml`
   works anywhere you have the group key. No vals, no Nix, no stackpanel CLI.

2. **vals is optional** -- Only needed for group key resolution from external
   stores (`.vals` files) or when a decrypted value itself is a vals reference
   (e.g., `ref+awsssm://...` inside a SOPS file).

3. **One reference, not many** -- Instead of N `ref+sops://` values in
   config.nix, a single `.vals` file per group resolves the group private key.
   Everything else is plain SOPS.

4. **Generate forward-compatible artifacts now** -- Devshell entry generates
   the v2 scripts, manifests, and configs. Existing `ref+sops://` consumers
   keep working. Migration of consumers happens later.

5. **Sensible defaults** -- `team` and `admins` recipient groups out of the
   box. `common` vars file for shared non-sensitive config.

## Directory Structure

```
.stackpanel/secrets/
├── .sops.yaml                    # Root SOPS config (generated by Nix)
├── groups.json                   # Recipient group → vars file mapping
├── vars/
│   ├── .sops.yaml                # Per-group creation rules (generated, gitignored)
│   ├── common.sops.yaml          # Shared config (encrypted to all group keys)
│   ├── dev.sops.yaml             # Dev secrets
│   ├── staging.sops.yaml         # Staging secrets
│   └── prod.sops.yaml            # Prod secrets
├── recipients/
│   ├── .sops.yaml                # For .enc.age files (generated, gitignored)
│   ├── team/                     # Default: everyone
│   │   ├── alice.age.pub         # Native AGE key
│   │   └── bob.ed25519.pub       # SSH ED25519 key (converted to AGE at build)
│   ├── admins/                   # Default: restricted access
│   │   └── alice.age.pub
│   ├── dev.age                   # Group private key, plaintext (gitignored)
│   ├── dev.enc.age               # Group private key, SOPS-encrypted (committed)
│   ├── dev.vals                  # Optional: vals reference for group key
│   ├── staging.enc.age
│   ├── prod.enc.age
│   └── .archive/                 # Rotated keys
(.stack/secrets/bin/ contains:)
├── join.sh                   # Add your key to a recipient group
├── rekey.sh                  # Re-encrypt .enc.age for current recipients
├── decrypt-env.sh            # sops decrypt → dotenv for a group/app
└── rotate.sh                 # Generate new group keypair + rekey
└── state/                        # Generated at shell entry (gitignored)
    ├── manifest.json             # Apps → groups → keys mapping
    └── sops-age-key-cmd.sh       # Generated SOPS_AGE_KEY_CMD script
```

## Default Recipient Groups

Two recipient groups are provided by default:

| Group | Access | Directory |
|-------|--------|-----------|
| `team` | `dev`, `common` | `recipients/team/` |
| `admins` | `dev`, `staging`, `prod`, `common` | `recipients/admins/` |

Mapping is declared in `groups.json`:

```json
{
  "team": {
    "vars": ["dev", "common"]
  },
  "admins": {
    "vars": ["dev", "staging", "prod", "common"]
  }
}
```

The `.sops.yaml` generation reads `groups.json` and collects all `.age.pub` /
`.ed25519.pub` files from each group directory to build creation rules.

## SSH Key Support

Recipients can use SSH ED25519 keys alongside native AGE keys:

```
recipients/team/
  alice.age.pub          # age1abc...
  bob.ed25519.pub        # ssh-ed25519 AAAA...
```

At `.sops.yaml` generation time:
- `*.age.pub` files: public key used directly
- `*.ed25519.pub` files: converted to AGE via `ssh-to-age`
- `*.rsa.pub` files: rejected with error (age does not support RSA)

The `.enc.age` files are encrypted to the AGE-equivalent keys of all
recipients across all groups (deduplicated).

## Group Key Resolution

When SOPS needs to decrypt a vars file, it calls `SOPS_AGE_KEY_CMD` to get
the group private key. The resolution chain:

```
1. Local AGE key (always)
   → .stackpanel/state/keys/local.txt

2. Plaintext group keys (if present, gitignored)
   → recipients/<group>.age

3. .vals files (if present, needs vals)
   → vals eval "$(cat recipients/<group>.vals)"
   → returns AGE private key

4. .enc.age files (default, needs sops + local key)
   → sops --decrypt recipients/<group>.enc.age
   → returns AGE private key
```

### `.vals` files

A `.vals` file contains a single vals reference that resolves to an AGE
private key:

```
# recipients/dev.vals
ref+awsssm:///stackpanel/keys/dev
```

This replaces the per-value `ref+sops://` pattern with a single reference per
group. Benefits:
- 1 vals resolution per group instead of N per secret
- vals becomes optional (teams using only `.enc.age` don't need it)
- External key stores (AWS SSM, 1Password, Vault) work via standard vals

## `common.sops.yaml`

The `common` vars file is encrypted to all **group keys** (not individual
recipients). This means anyone who can decrypt any group's `.enc.age` file can
also decrypt `common`:

```
Decryption chain for common:
  your local key → dev.enc.age → dev private key → common.sops.yaml
```

The `.sops.yaml` creation rule for common lists the AGE public keys of all
configured groups (dev, staging, prod).

Use `common` for shared config that should be accessible to all team members
regardless of their specific group membership, but shouldn't be plaintext in
git. Plaintext values within common use the `# safe` SOPS comment marker.

## Shell Hook Changes

### Current behavior (v1)

1. Auto-generate local AGE keypair if missing
2. Auto-register local pub key as recipient in `recipients/<username>.age.pub`
3. Rebuild `recipients/.sops.yaml` from all `.age.pub` files
4. Generate `vars/.sops.yaml` from Nix group config
5. Export `SOPS_AGE_KEY_CMD`

### New behavior (v2)

1. Auto-generate local AGE keypair if missing (first time only, unchanged)
2. **Do NOT auto-register as recipient** -- the user's key is NOT copied to
   `recipients/`. Their key stays in `.stackpanel/state/keys/`.
3. **Check decrypt access**: attempt to decrypt `recipients/dev.enc.age` with
   the local key. If it fails:
   ```
   ╭─ Secrets ──────────────────────────────────────────────────╮
   │  Your key is not yet registered as a team recipient.       │
   │                                                            │
   │  To join the team, run:                                    │
   │    stackpanel secrets join                                 │
   │                                                            │
   │  This will add your public key to recipients/team/ and     │
   │  push it. A teammate or CI will rekey to grant access.     │
   ╰────────────────────────────────────────────────────────────╯
   ```
4. Generate `vars/.sops.yaml` from Nix group config (unchanged)
5. Generate `recipients/.sops.yaml` from all `.age.pub` + `.ed25519.pub` files
   (updated for SSH key support)
6. Export `SOPS_AGE_KEY_CMD` with `.vals` fallback support
7. **Generate v2 artifacts** in `.stackpanel/secrets/state/`:
   - `manifest.json`: app → group → key mapping derived from variables config
   - `sops-age-key-cmd.sh`: the generated key resolution script

### `stackpanel secrets join`

Interactive command that:

1. Detects the user's key type (AGE from `state/keys/local.pub`)
2. Determines username (git config → whoami)
3. Copies pub key to `recipients/team/<username>.age.pub`
4. Commits and pushes (with confirmation)
5. Tells the user to wait for rekey workflow or ask an admin to rekey

## GitHub Actions Workflow

### Design: thin wrapper

The workflow should be a thin shell that delegates to a script. All
implementation lives in `bin/rekey.sh` or the stackpanel CLI.

### Triggers

```yaml
on:
  push:
    paths:
      - '.stackpanel/secrets/recipients/**/*.age.pub'
      - '.stackpanel/secrets/recipients/**/*.ed25519.pub'
  workflow_dispatch:
    inputs:
      public_key:
        description: 'AGE or SSH ED25519 public key (age1... or ssh-ed25519 ...)'
        required: true
      username:
        description: 'Team member name (used for filename)'
        required: true
      group:
        description: 'Recipient group (team or admins)'
        default: 'team'
```

### Workflow body

```yaml
jobs:
  rekey:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4

      - name: Install tools
        run: |
          # Install age + sops + ssh-to-age (versions pinned)
          # Or: nix profile install nixpkgs#{age,sops,ssh-to-age}

      - name: Add key (manual dispatch only)
        if: github.event_name == 'workflow_dispatch'
        run: |
          .stack/secrets/bin/add-recipient.sh \
            --key "${{ inputs.public_key }}" \
            --name "${{ inputs.username }}" \
            --group "${{ inputs.group }}"

      - name: Rekey
        env:
          # Group private keys from GitHub Actions secrets
          SECRETS_AGE_KEY_DEV: ${{ secrets.SECRETS_AGE_KEY_DEV }}
          SECRETS_AGE_KEY_STAGING: ${{ secrets.SECRETS_AGE_KEY_STAGING }}
          SECRETS_AGE_KEY_PROD: ${{ secrets.SECRETS_AGE_KEY_PROD }}
        run: .stack/secrets/bin/rekey.sh

      - name: Commit and push
        run: |
          git config user.name "stackpanel[bot]"
          git config user.email "stackpanel[bot]@users.noreply.github.com"
          git add .stackpanel/secrets/recipients/
          git diff --staged --quiet || git commit -m "rekey: update recipients"
          git push
```

### `secrets/bin/add-recipient.sh`

Detects key type from prefix, writes the `.pub` file:

```bash
#!/usr/bin/env bash
# Detect key type and write to correct location
case "$KEY" in
  age1*)   EXT="age.pub" ;;
  ssh-ed25519*) EXT="ed25519.pub" ;;
  ssh-rsa*) echo "Error: RSA keys not supported by age"; exit 1 ;;
esac
echo "$KEY" > "recipients/${GROUP}/${NAME}.${EXT}"
```

### `secrets/bin/rekey.sh`

Collects all recipient pub keys (converting SSH keys via `ssh-to-age`),
re-encrypts all `.enc.age` files to the full recipient set:

```bash
#!/usr/bin/env bash
# Collect all AGE public keys from all recipient groups
# Convert .ed25519.pub via ssh-to-age
# For each .enc.age: decrypt with group key from env, re-encrypt to all recipients
```

## Gitignore via Files Module

Currently, secrets gitignore entries are added via
`stackpanel.files.entries.".gitignore".lines`. This is already the correct
approach (block-managed `.gitignore`).

Changes needed:
- Add `recipients/*.age` (plaintext private keys)
- Add `recipients/.archive/`
- Add `.stackpanel/secrets/state/` (generated v2 artifacts)
- Negate `!recipients/*.age.pub` and `!recipients/*.enc.age`
- Negate `!recipients/**/*.ed25519.pub`
- Add `vars/.sops.yaml` (generated, gitignored)

## Codegen

### Current approach (keep for compatibility)

Env-style codegen: Nix generates typed TypeScript modules in
`packages/gen/env/` with `process.env.KEY` access patterns. This
continues to work via the existing `ref+sops://` variable values.

### New approach: sops decrypt + quicktype

Available as an alternative codegen mode. The pipeline:

```bash
# Extract app-specific vars as JSON
sops decrypt vars/dev.sops.yaml --output-type json > .stackpanel/state/vars/dev.json

# Generate typed module (any language)
quicktype -l typescript -o gen/vars/dev.ts --src-lang json < .stackpanel/state/vars/dev.json
quicktype -l go -o gen/vars/dev.go --src-lang json < .stackpanel/state/vars/dev.json
quicktype -l python -o gen/vars/dev.py --src-lang json < .stackpanel/state/vars/dev.json
```

This is NOT run at devshell entry (requires decryption). It's an on-demand
command: `stackpanel secrets codegen` or `secrets:codegen`.

### Devshell entry codegen (v2 artifacts)

At shell entry, WITHOUT decrypting anything, generate:

1. **`state/manifest.json`**: Derived from Nix variables config.

   ```json
   {
     "groups": {
       "common": { "recipients": ["team", "admins"], "keyedTo": "group-keys" },
       "dev": { "recipients": ["team", "admins"] },
       "staging": { "recipients": ["admins"] },
       "prod": { "recipients": ["admins"] }
     },
     "apps": {
       "web": {
         "groups": ["dev", "common"],
         "keys": {
           "dev": ["DATABASE_URL", "API_KEY", "OPENAI_API_KEY"],
           "common": ["LOG_LEVEL"]
         }
       },
       "server": {
         "groups": ["dev", "common"],
         "keys": {
           "dev": ["DATABASE_URL", "REDIS_URL"]
         }
       }
     }
   }
   ```

2. **`state/sops-age-key-cmd.sh`**: The generated key resolution script
   with `.vals` fallback support.

3. **Wrapper scripts** added to devshell PATH:
   - `secrets:load <group>` → `sops decrypt vars/<group>.sops.yaml --output-type dotenv`
   - `secrets:load-app <app>` → loads relevant groups, filters to app keys
   - `secrets:load-json <group>` → `sops decrypt ... --output-type json`
   - `secrets:join` → add local key to team recipients
   - `secrets:codegen [--lang ts|go|py]` → quicktype pipeline
